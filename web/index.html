4.2 Descripción del problema

Explicación general y objetivos: Este proyecto implementa una simulación que modela la llegada y embarque de vehículos en una terminal de ferrys. El objetivo es gestionar colas, priorizar emergencias, respetar capacidades de vehículos y peso, y generar reportes y estadísticas de operación.

Contexto y propósito: Herramienta académica para evaluar políticas de carga, priorización y tiempos en un sistema de transporte por ferry, útil para análisis y validación de reglas de negocio.

4.3 Análisis de la solución

Organización y almacenamiento de datos

Los datos se representan mediante `struct` en C. Las colas se implementan con arreglos circulares estáticos que contienen elementos tipo `VehiculoConEstado` (cada uno con `activo` y `timestamp`). La entrada se lee desde `proy1.in` y se carga en `cola_todos_vehiculos`.

Justificación de decisiones

  • Uso de arreglos estáticos para simplicidad y compatibilidad con entornos de práctica.
  • Política de prioridad para emergencias implementada para minimizar tiempo de respuesta.
  • Representación de tiempo en minutos internamente y formato hora militar en I/O para comparaciones sencillas.

4.4 Estructuras de datos

Structs principales:

  • Vehiculo: campos de entrada y campos derivados (`tipo_vehiculo`, `es_emergencia`, `peso_toneladas`, `total_pasajeros`).
  • Ferry: identificador, capacidades, tiempos, estado y estadísticas.
  • VehiculoConEstado: `Vehiculo` + `activo` + `timestamp_asignacion`.
  • ColaVehiculos: arreglo circular con `frente`, `final`, `cantidad`.
  • Simulacion: estado global con `ferrys[]`, colas, orden de carga y métricas.

Estructuras aplicadas: arreglos estáticos y buffers circulares; contadores para estadísticas.

while(simulación no terminada):

    if sim.tiempo_carga_restante > 0: sim.tiempo_carga_restante--

Políticas clave

  • Prioridad emergencias: emergencias cargan primero; si no caben, se bajan vehículos normales para liberar espacio.
  • Condición de zarpe: mínimo 30% vehículos o condiciones de llenado, verificación de peso y pasajeros.

4.6 Pruebas realizadas

Casos: archivos de emergencias (ráfagas), archivos grandes (estrés), casos inválidos para validación de parser.
